# Sleipnir — Wire contracts

This document is the **frozen specification** for the Kafka payloads sleipnir consumes and produces. Changes to either type are coordinated cross-repo PR pairs with [huginn](https://github.com/lgreene03/huginn).

## Topics

| Topic | Direction | Producer | Consumer | Schema |
|---|---|---|---|---|
| `executions.intents.v1` | huginn → sleipnir | huginn `internal/kafka/producer.go` | sleipnir `internal/gateway/gateway.go` | `Order` (below) |
| `executions.fills.v1` | sleipnir → huginn | sleipnir `internal/kafka/producer.go` | huginn `internal/kafka/fills_consumer.go` | `ExecutionFill` (below) |

Topic names carry a `.v1` suffix. Breaking changes ship a new topic (`v2`); the payloads themselves do not carry a version field today.

## `Order` (inbound intent)

Defined in sleipnir at `internal/exchange/connector.go`, decoded by `internal/gateway/gateway.go`. Encoded as JSON.

Huginn's matching type lives at `huginn/internal/kafka/producer.go:14` as `GatewayOrder`.

```json
{
  "orderID":    "huginn-1716123456789",
  "instrument": "BTC-USD",
  "side":       "BUY",
  "type":       "MARKET",
  "quantity":   0.001,
  "price":      0.0,
  "timestamp":  "2026-05-19T18:30:00Z"
}
```

| Field | Type | Notes |
|---|---|---|
| `orderID` | string | Generated by huginn. Used as Binance `newClientOrderId`. **Validation hardening tracked in audit H4.** |
| `instrument` | string | Canonical form `BASE-QUOTE`, e.g. `BTC-USD`. Sleipnir translates to Binance exchange form `BTCUSDT`. |
| `side` | string | `BUY` or `SELL`. |
| `type` | string | `MARKET` or `LIMIT`. |
| `quantity` | float64 | Base asset units. |
| `price` | float64 | Required for `LIMIT`; 0 for `MARKET`. |
| `timestamp` | string | RFC3339. Producer's wall-clock at intent generation. |

## `ExecutionFill` (outbound fill)

Defined in sleipnir at `internal/exchange/connector.go`. Huginn's matching type lives at `huginn/internal/kafka/fills_consumer.go:14` as `GatewayFill`.

```json
{
  "order_id":         "huginn-1716123456789",
  "execution_id":     "huginn-1716123456789-ws-987654321",
  "instrument":       "BTC-USD",
  "side":             "BUY",
  "quantity":         0.001,
  "fill_price":       43520.10,
  "transaction_cost": 0.04352,
  "timestamp":        "2026-05-19T18:30:01.234Z"
}
```

| Field | Type | Notes |
|---|---|---|
| `order_id` | string | Echoes the inbound intent's `order_id`. |
| `execution_id` | string | Deterministic per-fill identity. Used by huginn for cross-restart deduplication. See construction patterns below. |
| `instrument` | string | Canonical form (same as intent). |
| `side` | string | `BUY` or `SELL`. |
| `quantity` | float64 | Delta quantity for this fill (not cumulative). Today's gateway emits one event per WS `executionReport`. |
| `fill_price` | float64 | Trade price. |
| `transaction_cost` | float64 | Commission in quote-currency units. **Currently 0.0 from the REST submit path and the reconciliation backfill** — audit L6, remaining Phase 5 work. |
| `timestamp` | string | RFC3339. Should be the exchange transaction time. Today's reconciliation path uses `time.Now()` — remaining Phase 5 work. |

### `execution_id` construction

Three codepaths produce fills, each with a stable construction pattern. No two real fills across the three paths can collide.

| Source | Pattern | Example |
|---|---|---|
| REST submit response (`SubmitOrder`) | `<order_id>-rest-<binance order id>` | `huginn-1716123456789-rest-12345` |
| WebSocket `executionReport` | `<order_id>-ws-<binance trade id>` | `huginn-1716123456789-ws-987654321` |
| WebSocket fallback (trade id absent) | `<order_id>-ws-<transact_time_ms>-<qty_string>` | `huginn-1716123456789-ws-1716123456789-0.00100000` |
| Boot reconciliation backfill | `<order_id>-reconcile-<delta_qty>` | `huginn-1716123456789-reconcile-0.001` |

Huginn's `executor.OnExecutionFill` keeps a bounded LRU (10 000 entries) of seen `execution_id` values and drops duplicates with a WARN log. This is what makes the cross-restart reconciliation path safe.

## Wire-format change protocol

1. **Open a discussion** in the sleipnir repo describing the change and the migration path.
2. **Update this document first** (in a draft PR) to describe the new field and its semantics.
3. **Land the producer side first**, with the new field optional and defaulted on the consumer side, so old consumers still parse.
4. **Land the consumer side**, gated by a feature flag or version field if the new field changes semantics.
5. **Remove the old code path** in a follow-up PR after both repos are deployed.

For breaking changes: ship a new topic version (`executions.intents.v2`), keep `v1` consumers running until the cutover is complete, then retire `v1`.

## Field-rename hazard

A field rename in either `Order` or `ExecutionFill` is **silently destructive**: JSON ignores unknown fields by default, so the consumer happily reads zero values. The Phase 4 contract test (sleipnir decodes a recorded huginn `GatewayOrder` JSON blob from `testdata/`) catches this in CI within 24 hours of merge. Until that test exists, rename only via the protocol above.
